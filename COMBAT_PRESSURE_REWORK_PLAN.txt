BEHAVIOUR (all tuning in Balance::Hazards):
- FIRE TOTEM: every 3.2s raises a red aim line for 0.9s that tracks the player,
  FREEZES for the last 0.35s (dodge window), then fires one EnemyProjectile
  FireBolt (1 dmg). 3 player hits destroy it; each hit also disables it 4s and
  cancels a mid-aim.
- LAVA POOL: animated pool sprite, damage circle r=100 inside it, 0.45s grace
  on first touch, then 1 dmg per 0.5s tick. Not destructible. Hot/underground
  biomes only (Caverns / DemonsInsides / Wastelands).
- FIREBALL TORCH: glow-orb emitter mounted on the east/west wall OUTSIDE the
  door band (kTorchDoorBandHalf 170px), fires a horizontal FireBolt lane every
  2.6s with a 0.5s pre-fire flash. 3 hits destroy / hit disables like the totem.
- All hazards: 1.1s telegraph ring on spawn (semi-transparent sprite + pulsing
  circle), plus a 1.5s room-entry grace before the first action. Environmental
  FireBolt cap (10) shared with Flame Wisps — counted per frame in Engine.

PLAYER COUNTERPLAY WIRING:
- Melee swings (HandlePlayerMeleeDamage), every ability rect
  (DamageEnemiesInRect), and player projectiles (UpdateSpreadProjectiles —
  bolts burst on hazards) all call _roomHazards.DamageHazardsInRect.
  Hazards count HITS (0.25s per-hazard hit cooldown), not scaled damage.

GENERATION (Phase-6 first slice): standard combat rooms roll a hazard at
kRoomFrequencyByTier {38/58/72%}; placement validated (doors/entry/spacing/
edge margin, 8 attempts); hazard pressure cost (2) is deducted from the room's
enemy pressure budget. Boss/Elite/Treasure/Store/Rest rooms never roll.

KNOWN LIMITS (deliberate, revisit later): hazards don't respawn when
re-entering a half-cleared room (they only exist on first visit); no
destruction VFX/sound yet; deterministic per-room seeding still Phase 6.

PHASE 1 — AUDIT OF EXISTING SYSTEMS (what already exists, where)
--------------------------------------------------------------------------------

ENEMY UPDATE FLOW
- CombatDirector::UpdateEnemyRuntime(EnemyRuntimeContext, dt) is the single
  enemy tick loop (called from two Engine sites: old wave mode ~Engine.cpp:2300
  and DungeonRun ~Engine.cpp:16940). Context carries player, nav, propCenters,
  projectile lists, spawn callbacks, and (new, 2026-07-12) hazards = the
  player-made damage-zone list enemies steer around.
- Enemy::HandleMovement already does steering: waypoint pull + approach-slot
  fan-out + enemy separation (130px) + prop repulsion w/ tangential slide
  (110px) + NEW hazard repulsion. Any new "avoid X" behaviour belongs here.

ENEMY / BOSS PROJECTILES (reuse these, do NOT duplicate)
- EnemyProjectile (EnemyProjectile.h): generic straight shot, kinds
  Arrow / FireBolt (animated 32px sheet) / Spit. Engine owns the list
  (_enemyProjectiles); collision centralized with lava balls + cyclops lasers.
  ==> Fire Totem and Fireball Torch shots should be EnemyProjectile::FireBolt
      (or a new kind TotemBolt if different speed/scale is needed).
- LavaBallProjectile: elite "Hazards" mechanic volley (mechanic 4 in elite
  rooms, kHazardVolley* constants) — arcing AoE with warning shadow.
- CyclopsLaserProjectile: aimed/scatter boss laser.

EXISTING ENVIRONMENTAL HAZARDS (biome modifiers, Engine.cpp ~15954)
- Wastelands: WastelandHazard — random warning circle (DrawCircleLines pulse)
  -> Flame_Explosion.png sprite burst, 0.3s damage tick, kWasteland* consts.
- LostCity: _lostCityBeams — rotating beams, segment-distance damage test,
  cooldown per beam, drawn with DrawLineEx (NO owned VFX yet).
- TheSanctuary: _sanctuaryZones — dash-lock + slow discs (player debuff only).
- DemonsInsides: _demonPulseTimer pulse attack.
- Boss poison clouds: SpawnPoisonCloud(pos, radius) + VFXManager hazard decals
  (VFXManager::SpawnHazardDecal) — the FX-sprite ground-zone pipeline exists.
==> These are per-biome one-offs switch-cased in UpdateBiomeModifiers /
    DrawBiomeModifiers. The RoomHazardDirector should NOT rip them out
    immediately; wrap/migrate only where it clearly removes duplication
    (candidate: Wasteland explosion == "eruption crack" hazard template).

ENCOUNTER COMPOSITION (SpawnDungeonRoomEnemies, Engine.cpp ~13785)
- Tier = cleared-room count (0/1/2) drives min/max basics (2-3 / 3-5 / 4-6)
  + zone bonus (_worldZone/2) + DemonsInsides +5 + Swarm affix multiplier.
- Weighted type table (10 enemy types), Warchief capped 1/room.
- EnemyRole enum already exists (Grunt/Charger/Ranged/Tank/Support/Assassin/
  Zoner/Summoner/HeavyRanged/Boss) with GetEncounterRole() per enemy and
  role-based placement (GetDungeonSpawnPosForRole).
==> The pressure budget slots in here: role -> pressure cost table, room cap
    by tier, and reinforcement waves when desired population > readable cap.
- Room state: _dungeonRoomStates (cleared / enemiesInitialized / survivors)
  — hazards must serialize alongside if they persist across re-entry
  (simplest: regenerate deterministically from room index seed).

DIRECTIONAL DEFENCE (pre-Phase-2 state, now FIXED)
- Shieldbearer::TakeDamage and TitanGuard::TakeDamage used a raw x-sign
  half-plane test (attackSide * _rightLeft > 0), and base Enemy code flipped
  _rightLeft toward the player every frame -> rear positioning near-impossible.
- Rogue Backstab was DamageEnemiesInRect with big damage — no rear check.

DEBUG / TELEMETRY INFRA
- _debug (DebugPanel, F1/F10 toggle) — IsActive()/IsOpen(); RoomTelemetry ring
  drawn beside it. Debug visuals hang off the _debug.IsOpen() block
  (~Engine.cpp:17816). TraceLog available for logging.

TUNING HOME
- GameBalance.h namespaces (Balance::Grunt/Elite/Boss/Sustain/Rooms/...).
  New namespaces added: Balance::Facing (Phase 2). Planned: Balance::Pressure,
